Seven issues reported from a real run (beats 06 → 12, EASY, Heavy Mallet).
Every one has been traced to code by reading, not guessing. **Nothing here is
fixed yet.** This document is the handoff.

Read `HANDOFF.md` first for the project's standing traps. The three that bite
hardest on this batch:

- **Measure before you tune.** Three of six numbers in an earlier ticket were
  wrong when finally measured. Write a throwaway probe in `tests/qa/_tmp-*.mjs`,
  get the real number, delete the probe.
- **Sweep every place, not one.** This project's most expensive recurring bug is
  fixing the one site that was reported and leaving three others. Issues 4, 6
  and 7 below each have more than one site.
- **Assert directions in world space,** never the sign of a rotation angle.

**Pushing:** `origin` in this repo is **My-Engine**, not the game. A plain
`git push` publishes the game to the wrong repository and prints a normal,

## 1 — Breakable minerals pay nothing

> *"What is the point of the minerals I break?"* — and, on being told they do
> nothing: **"Why not all three"** (drop loot, reveal secrets, gate progress).

### Status

Confirmed. There is no reward path of any kind.

### What exists

`DestructibleVoxelMesh` ([destructible-voxel-mesh.js](../src/game/world/destructible-voxel-mesh.js))
is a complete, well-built system: a voxel map that is truth, a mesh that is a
view, BFS shatter that stops at colour boundaries, collision solids rebuilt on
every change. It works.

`Player.tryAttack` drives it at
[player.js:200-211](../src/game/player.js#L200):

```js
if (weapon.shatter && destructibles) {
    for (const d of destructibles) {
        const n = d.shatterAtWorld(px, py, pz, weapon.shatterRadius || 3);
        if (n > 0) vsfx.shatter();     // <- the entire consequence
    }
}
```

`shatterConnected` **returns the number of voxels removed** and every caller
throws it away except to decide whether to play a sound. That return value is
the hook the reward would hang on, and it is already there.

Boulders are placed by `addBoulders` at
[beat-06-quarry.js:24-39](../src/game/levels/beat-06-quarry.js#L24) — three in
`quarryfloor`, two in `orecrush`. They are pushed onto `level.destructibles` and
nothing else ever reads that array except the shatter loop above.

### Why it feels worse than "no reward"

The Quarry's *stated* dungeon item is the Heavy Mallet, toasted as
**"Heavy Mallet — shatter ore veins"** ([beat-06-quarry.js:126](../src/game/levels/beat-06-quarry.js#L126)).
The game explicitly promises that breaking ore is a verb worth having, then
gives it no object. The gold-vein walls that actually gate progress are
`blockers`, a different system entirely — so the item's real use never touches
the thing the toast pointed at.

### Recommended fix — all three, because they are three different jobs

Do them in this order; each is independently shippable.

**(a) Loot.** The drop machinery already exists and is already correct — Z5 built
`dropSite(enemy)` and `HeartDropManager.dropLoot` for the flying-enemy bug. A
destructible is not an enemy but it has a position, so give
`heart-drops.js` a `dropAt(x, y, z, table)` and call it from the shatter site
when `n > 0`. Weight it low (shards mostly, a heart occasionally) — the point is
that breaking things is *sometimes* worth it, not that it is a farm.

**Sweep both sites:** `player.js:200-211` is the melee path. Check whether
anything else shatters (bosses, the Sand Spur's burrow at
[sand-spur.js:103](../src/game/bosses/sand-spur.js#L103)) and decide per site
whether it should pay.

**(b) Reveals.** `level.addPickup` already takes a world position. Author a
`hidden` field on the boulder spec in `addBoulders`, and spawn its pickup at the
boulder's origin when the last voxel of that island goes. `isEmpty` is already a
getter on `DestructibleVoxelMesh` — that is the exact signal.

**(c) Gating.** This is the biggest of the three and the most Zelda. A boulder
sitting in a doorway that the room's clear-gate waits on turns "break the rock"
into a puzzle rather than scenery. Note the softlock hazard the project already
hit once with brood children spawning inside masonry
([enemy.js:797 `freeSpotNear`](../src/game/enemy.js#L797)): if a gate waits on a
destructible, prove the player cannot reach the gate without the tool that
breaks it, or you have built a save-ruining dead end.

### How to prove it

A spec that shatters a boulder headlessly and asserts `level.pickups.length`
increased, plus a counterfactual: revert the reward hook and assert it stays
flat. Add a Beat 06 clearability spec for (c) — no gate may depend on a
destructible the player cannot yet break.

## 2 — Beat 07 Sluice of Tears is blown out

> *"this level is FUCKING BRIGHT!"*

### Status

Confirmed. The screenshot shows a near-white floor with the gold checker inlay
barely readable against it.

### Root cause

Two multipliers stacking on a pale palette.

[beat-07-sluice.js:25](../src/game/levels/beat-07-sluice.js#L25):

```js
lightTune: { ambient: 2.0, key: 1.85 },
```

That is the second-brightest tune in the game. The full table, for context:

| Beat | ambient | key |
|---|---|---|
| 12 Pyre | 2.3 | 2.05 |
| 09 Town | 2.25 | 2.0 |
| **07 Sluice** | **2.0** | **1.85** |
| 06 Quarry | 1.8 | 1.45 |
| 14 Leviathan | 1.2 | 1.5 |

The renderer then applies ACES with exposure above 1
([renderer.js:49-50](../src/engine/renderer.js#L49)):

```js
renderer.toneMapping = THREE.ACESFilmicToneMapping;
renderer.toneMappingExposure = 1.25;
```

ACES does not hard-clip, it rolls off — which is why this reads as *desaturated
milk* rather than flat white. A pale floor pushed by `ambient 2.0 × key 1.85 ×
exposure 1.25` lands deep in the shoulder, where the tone curve compresses
everything into the same near-white and the floor's own colour information is
destroyed. The Sluice's kit
([dungeon-kits.js:70-74](../src/game/levels/dungeon-kits.js#L70)) adds a
`lanes` inlay at `capShade 1.3` on top, lifting it further.

### Why the certification gate did not catch it — this is the important part

The gate ([visual-sanity.spec.mjs:16-48](../tests/visual-sanity.spec.mjs#L16))
asserts exactly two things per level:

```js
const LUM_BANDS = { crust: [45, 90], abyss: [45, 90] };   // frame-wide MEAN
const CONTRAST_FLOORS = { dungeon: 60, open: 10 };        // centre-crop p90 − p10
```

**Neither statistic can see a locally blown-out floor.**

- The **mean** is over the whole frame. A floor at ~230 averaged against dark
  walls and the large unlit regions lands in the 50s — comfortably inside
  `[45, 90]`. `CERTIFICATION.md` records the Sluice at `53 / 42`.
- The **contrast floor** is `p90 − p10`, and a *checkerboard* has enormous
  p90 − p10 whether or not it is blown out. The inlay pattern that makes the
  room look washed out is the same thing that makes it pass the anti-flatness
  check.

This is the third time this project has shipped a gate that passes for the
wrong reason (see the truncated audio render and the original mean-luminance
trap, both in `HANDOFF.md`). The pattern is always the same: the statistic
measures something adjacent to what the eye objects to.

**The missing statistic is a clipping/shoulder fraction** — the proportion of
centre-crop pixels above ~245, or better, the proportion whose three channels
are within a few counts of each other *and* above ~200 (ACES desaturates as it
compresses, so "bright and grey" is the actual signature of the shoulder).

### Discrepancy to reconcile first

`CERTIFICATION.md:95` records the Sluice as `53 / 42`. If that second figure is
the centre-crop contrast, **42 is below the dungeon floor of 60 and the gate
should be failing right now.** The suite is reported green at 3013/3013, so one
of these is stale or the column means something else. Resolve this before
trusting either number — do not tune against a figure you have not re-measured.

### Recommended fix

1. **Measure first.** Capture the Sluice rooms the player was actually standing
   in (the screenshot is not the `entry` or `boss` capture) and record mean,
   p90 − p10, and the new clipping fraction.
2. **Add the clipping assertion** to `visual-sanity.spec.mjs` and prove it
   discriminates — a synthetic near-white frame must fail it while passing both
   existing checks. That proof is the whole value of the assertion; without it
   you have added a third number that also cannot see the problem.
3. **Then** bring `ambient`/`key` down until the new statistic passes, and check
   the mean has not fallen out of `[45, 90]` at the bottom.
4. **Sweep every place.** Beats 09 (2.25/2.0) and 12 (2.3/2.05) are *brighter*
   than the Sluice. The user's screenshots of both show the same washed look.
   Do not fix only the level that was named.
5. Regenerate the 44 certification captures afterwards
   (`tests/qa/certification-captures.mjs`) and update `CERTIFICATION.md`.

## 3 — Pickups are featureless white blobs

> *"What is the point of this shiny point?"*

### Status

Confirmed. Same root cause as #2, which is why they should be fixed together.

### Root cause

[pickup-shapes.js:18](../src/game/assets/pickup-shapes.js#L18):

```js
function mat(color, emissiveIntensity = 2, extra = {}) { ... }
```

with the Scar Suture at `2.4`. Under ACES at exposure 1.25, an emissive of 2.0+
sits well into the shoulder, where the curve compresses and desaturates — so the
pickup renders as a white ball with a faint colour fringe, and its silhouette is
gone.

### Why this is worse than a cosmetic complaint

The file's own header
([pickup-shapes.js:1-13](../src/game/assets/pickup-shapes.js#L1)) states the
design intent:

> Every pickup in the game … was the same 0.35 octahedron in a different colour.
> From a camera 17.5 units up, colour is the weakest signal there is … Shape
> survives all three.

Distinct silhouettes were authored *specifically* so a player could tell a shard
from a suture across a dark room. Then the emissive was set high enough to
destroy the silhouette. **The fix that was built and the setting that defeats it
are in the same file.** The player's question — "what is the point of this shiny
point" — is the exact failure the header predicted, arriving by a different
route than the one it guarded against.

### Recommended fix

Drop `emissiveIntensity` until the shape survives. Measure it: render each of
the pickup silhouettes over both a dark and a bright floor and assert that the
centre-crop of the pickup has non-trivial internal contrast — i.e. that it still
has *edges*. A pure white ball has none, which makes this cheaply testable.

Do **not** simply darken them until they vanish on a dark floor; the reason the
emissive is there is that pickups must be findable. The target is "glowing and
still shaped", so the assertion needs both a floor and a ceiling.

Consider also whether the contact-shadow disc under each pickup (Ticket 2) gives
enough grounding to let the emissive come down further than it otherwise could.

## 4 — Reflected bolts do not kill; ranged enemies still need a parry

> *"Brown ranged enemies still require a parry to die, no shooting enemy should
> require that, though blasting their own projectile back at them should kill
> them, which it currently does not."*

### Status

Confirmed by arithmetic. The mechanic works exactly as built; the number is
simply too small.

### Which enemy this is

The "brown ranged enemy" is a **lancer with `ai: 'ranged'`** — palette
`0x9a4a4a`, "long, thin, arterial red"
([palettes.js:188](../src/game/assets/palettes.js#L188)), which reads brown
under the Abyss grade. Two rooms in Beat 07 spawn one:
[beat-07-sluice.js:108](../src/game/levels/beat-07-sluice.js#L108) and
[:232](../src/game/levels/beat-07-sluice.js#L232).

### The arithmetic

Reflect damage, [enemy.js:20](../src/game/enemy.js#L20) and
[:724](../src/game/enemy.js#L724):

```js
const REFLECT_DAMAGE_MULT = 2;
...
p.damage *= REFLECT_DAMAGE_MULT * (player.guard?.parryReady ? 2 : 1);
```

Base projectile damage is the enemy's `damage`, default **1**
([enemy.js:83](../src/game/enemy.js#L83), [:630](../src/game/enemy.js#L630)).

- Held shield → **2 damage**
- Parried → **4 damage**

Enemy HP for that lancer, via `scaleEnemyHp(3, 7)`
([threat-curve.js:75](../src/game/world/threat-curve.js#L75)):

```
(3 / 4) × TARGET_HITS[7]=2.8 × playerDamageAt(7)=2.5  =  5.25 → 5.5
× mode.enemyHp (EASY) 0.70                            =  3.85
```

**So: a held-shield reflect does 2 against 3.85 HP — it survives on 1.85.
A parried reflect does 4 — it dies outright.**

That is the reported experience, exactly: hold the shield and the bolt bounces
back and the thing lives; time the parry and it dies. The player correctly
concluded a parry is required, because at these numbers it *is*.

It worsens with depth (same authored HP 3, EASY):

| Beat | scaled HP | held-shield reflects to kill | parried reflects to kill |
|---|---|---|---|
| 07 | 3.85 | 2 | 1 |
| 09 | 4.9 | 3 | 2 |
| 12 | 5.6 | 3 | 2 |

### Second defect in the same path — a reflected bolt can only ever hit its own shooter

[enemy.js:640-652](../src/game/enemy.js#L640). `_updateProjectiles` is a method
on the enemy that fired the bolt, and the reflected branch tests distance to
`this.rig.position` only. Reflecting into a group of three shooters can never
hit the other two. Whether that should change is a design call, but it is
currently invisible to the player and reads as the reflect "missing".

### Third — a reflected bolt ignores geometry entirely

No projectile in the game tests collision. A bolt sent back passes through
walls, and so does the original. Low priority, but it is the same loop.

### Recommended fix

Raise `REFLECT_DAMAGE_MULT` so a **held-shield** reflect is a genuine answer.
Setting it so that one clean reflect kills an ordinary shooter at the beat it
appears means scaling with the curve, not a constant — the curve triples enemy
HP across the campaign while `REFLECT_DAMAGE_MULT` stays at 2, which is why this
gets worse as you go.

The cleanest shape: make a reflected bolt's damage a **fraction of the shooter's
own max HP** rather than a multiple of its attack. "Your own shot kills you" is
then true at every beat by construction, and the parry bonus stays a bonus
rather than the price of admission.

**Keep the parry worth doing** — it should stay strictly better (faster, or it
also staggers, or it pierces) so the skill ceiling survives. The user's
objection is not that parrying is rewarded, it is that it is *mandatory*.

### How to prove it

Extend the four projectile scenarios already in
`tests/game/bestiary.spec.mjs`. The one to add: spawn the lancer at its Beat 07
scaled HP, reflect one bolt with `guard.raised` true and `parryReady` **false**,
and assert `state.current === 'DEAD'`. Run it against current code first — it
must fail — then fix.

## 5 — The flying mote cannot be reached at all

> *"Purple enemy can't be reached at all. Should be something I can beat on even
> in the air."*

### Status

Confirmed by arithmetic. It is unreachable **by design**, and the design is
wrong for how the game actually plays.

### The arithmetic

[enemy.js:137-142](../src/game/enemy.js#L137):

```js
this.hover = opts.hover != null ? opts.hover : this.kind === 'mote';
this.flyHeight = opts.flyHeight != null ? opts.flyHeight : 3.4;
```

with this comment, which is the whole story:

> 3.4 is not arbitrary: the tallest melee move (heavy_mallet) has vertical 1.5,
> plus a 0.5 hit radius, against a player rig sitting at 1.95. Anything under
> ~3.2 is still swingable, **which would quietly turn the mote back into an
> ordinary enemy.**

The vertical gate is [hitbox.js:53-54](../src/combat/hitbox.js#L53):

```js
const dy = defender.root.position.y - attacker.root.position.y;
if (Math.abs(dy) > move.vertical + r) return false;
```

Measured: mote at `y ≈ 4.4`, player at `y ≈ 1.45`, so `dy ≈ 2.95`. Heavy Mallet
`vertical 1.5` + mote `hitRadius 0.5` = **2.0**. `2.95 > 2.0` → every melee
swing in the game is rejected before it is even considered.

### The two answers that exist, and why neither reached the player

1. **The Light Caster** (ray weapon, `range 12`, no vertical gate on the ray
   path). The player was carrying the **Heavy Mallet** in all five screenshots.
2. **Parry its burst**, which calls `stagger()` → `_groundedT`
   ([enemy.js:271-286](../src/game/enemy.js#L271)) and drops it into reach. This
   is the "one rule, uniformly applied" the code is proud of — and it requires
   the player to already know that parrying a burst grounds a flyer, which the
   game never says.

So the mote's honest answer set is "switch weapons, or execute a mechanic
nobody told you about". That is not a question the bestiary is asking well.

### Recommended fix — a design decision is needed here

The user has stated the outcome they want: *"should be something I can beat on
even in the air."* That is a legitimate call and it is theirs to make. Options,
cheapest first:

**(a) Lower `flyHeight` into melee reach.** One number. Costs the kind its
identity — the code comment above says so explicitly, and it is right.

**(b) Make the mote descend on a cycle.** It drops to swingable height to wind
up its burst, and climbs back after. The burst becomes a *window* rather than a
wall, melee gets a real answer, and the kind keeps "you have to time it, not
just walk up to it". This is the most Zelda of the three and the recommended
one.

**(c) Give melee an upward reach.** Raise `vertical` on heavy weapons, or add an
explicit anti-air. Broadest blast radius — `vertical` is read by every melee
check in the game, so changing it changes every fight. Only do this with the
full suite as a guard.

Whichever is chosen, **teach it.** A `coach()` hint on first sight of an
airborne mote costs one line and is the difference between a mechanic and a
wall. `coach('armor-front', ...)` at [enemy.js:171](../src/game/enemy.js#L171)
is the pattern to copy.

### Note while you are in here — the `bulwark` kind is never spawned

`grep "kind: 'bulwark'" src/game/levels/` returns **nothing**. No level in the
campaign spawns one. The README advertises it ("the bulwark's front plate must
be flanked or parried") and `tests/game/bestiary.spec.mjs` tests it, so it is
green, documented, and unreachable in play. Out of scope for this batch, but
worth a ticket.

## 6 — Bosses walk through walls

> *"Boss went through the wall I can't hit him. Finally hit him by going around
> the back, but still should not clip."*

### Status

Confirmed. **This is the blocker of the seven.** Bosses have no collision of any
kind.

### Root cause

`BossBase`'s constructor is `(scene, opts)`
([base.js:27](../src/game/bosses/base.js#L27)). **It never receives a
`collisionWorld`.** There is no reference to one anywhere in the file — a boss
has no way to consult geometry even if it wanted to.

Contrast with `Enemy`, which does it correctly
([enemy.js:743-759](../src/game/enemy.js#L743)):

```js
_move(dx, dz, dist, sp) {
    if (this.collisionWorld) {
        const r = this.collisionWorld.resolveMove(x0, z0, nx, nz, 0.4);
        this.rig.position.x = r.x;
        this.rig.position.z = r.z;
    } else { /* ... */ }
}
```

The boss movement helpers write straight to the transform with no check at all.
[base.js:681-701](../src/game/bosses/base.js#L681):

```js
export function circleStrafe(pos, player, dt, opts = {}) {
    ...
    pos.x += (ddx / dd) * step;
    pos.z += (ddz / dd) * step;
}
```

and [base.js:704-711](../src/game/bosses/base.js#L704):

```js
export function moveToward(pos, target, speed, dt) {
    pos.x += (dx / d) * speed * dt;
    pos.z += (dz / d) * speed * dt;
}
```

The Phantasm — **the Beat 09 boss the user was fighting** — calls both, back to
back, every frame ([roster.js:851-853](../src/game/bosses/roster.js#L851)):

```js
moveToward(this.root.position, target, 2.5, dt);
circleStrafe(this.root.position, player, dt,
    { speed: 2.2, spin: 0.9, close: 1.4, minRadius: 1.6 });
```

### Why "I had to go around the back"

`circleStrafe` spirals *inward toward the player* — its radius only ever shrinks
(that is deliberate, and documented: a boss holding a fixed radius from a
chasing player is unreachable). So when the player backs toward a wall, the boss
drives at them and, having nothing to stop it, ends up **inside or behind the
masonry**. Its centre is then unreachable from the front — the player's swing
resolves against a position on the far side of a solid — and the only way to
land a hit is to walk around to whatever side the body is actually on. Which is
what the user did.

### A third helper exists and would have prevented this

[base.js:714-728](../src/game/bosses/base.js#L714) exports `bounceArena(pos,
vel, center, radius)`, which clamps a position inside an axis-aligned box.
Nothing calls it. Every boss already has `this.home`
([base.js:98-101](../src/game/bosses/base.js#L98)) — the arena centre it was
placed at. The clamp was built, the anchor it needs was built, and they were
never connected.

### Recommended fix

Two layers, and **do both** — either alone leaves a hole.

**(a) Real collision.** Thread `collisionWorld` into `BossBase` and add a
`_move(nx, nz)` that routes through `resolveMove` exactly as `Enemy._move` does.
Then change `circleStrafe` and `moveToward` to go through it rather than writing
`pos` directly. Use the boss's `hitRadius` as the collision radius, not the
`0.4` the enemies use — bosses are `presenceScale`d up to 2× and a 0.4 probe on
a 2.2-radius body still buries most of it in the wall.

**(b) An arena clamp as a backstop.** Wire `bounceArena` (or a simple clamp) to
`this.home` plus the room's half-extent. Collision resolution can still be
tunnelled through by a fast leap — the Arachnid's pounce moves further in one
frame than a wall is thick — and a hard clamp is the cheap guarantee that a boss
is *always* somewhere the player can stand.

**Sweep every place.** `moveToward` and `circleStrafe` are the shared helpers,
but several bosses write `this.root.position` directly in their own `tickAI` —
e.g. [roster.js:695-696](../src/game/bosses/roster.js#L695),
[:1092](../src/game/bosses/roster.js#L1092). Grep `root.position` across
`bosses/` and fix all of them, or you will fix the Phantasm and ship the same
bug in the Magma Wyrm. This is the exact failure mode recorded in
`sweep-every-place-not-one`.

### How to prove it

`tests/game/bosses.spec.mjs` already has the harness for this. Add: place a
solid at a known position, drive the boss at it for two seconds of simulated
time, and assert the boss's final position is **outside** the solid's bounds.
Assert in world space, not by checking a flag. Run it before fixing — it must
fail.

## 7 — Enemies and bosses leave the room through doorways

> *"Boss and enemies can go outside the room through the door, this should be
> fixed."*

### Status

Confirmed. Two different causes for the two cases — do not assume one fix covers
both.

### Enemies

`Enemy._move` *does* respect walls
([enemy.js:747-754](../src/game/enemy.js#L747)) — that is why they do not clip
through masonry. But **a doorway is a genuine gap in the collision geometry**,
and nothing anywhere confines an enemy to the room it was baked into. An enemy
chasing the player simply walks out of the hole in the wall, because from the
collision world's point of view there is nothing there to stop it.

Rooms are baked at [room-graph.js:308-315](../src/game/world/room-graph.js#L308):

```js
for (const e of room.enemies || []) {
    const enemy = new Enemy(scene, collisionWorld,
        { x: origin.x + e.x, y: 1.0, z: origin.z + e.z },
        { ...e, hp: scaleEnemyHp(e.hp, beatNo) });
```

The enemy gets a world position and a collision world. It is **never told which
room it belongs to**, and no update-time check puts it back.

`rec.enemies` per-room bookkeeping does exist and `disposeRoom` cleans them up
([room-graph.js:330-340](../src/game/world/room-graph.js#L330)) — so a wandered
enemy is still freed correctly. The bug is behavioural, not a leak.

### Bosses

Same root as issue 6 — no collision at all, so a boss does not even need the
doorway. Fixing 6 fixes the boss half of this **only if you do the arena clamp**
(6b); collision alone still lets a boss walk out through an open door.

### Recommended fix

Give each enemy the bounds of the room it was baked into (`origin` and `half`
are both in hand at the bake site) and clamp `_move`'s result to them. Two
details that matter:

- **Clamp, do not reflect.** An enemy that bounces off an invisible wall in a
  doorway looks broken in a different way. Clamping to the room's interior reads
  as "it will not follow you out", which is the Zelda convention and what the
  player expects.
- **Leave the player's own door transit alone.** The clamp is on the enemy's
  position, not on the doorway's collision — do not plug the gap, or the player
  cannot walk through it either.

Check `brood` split children specifically: `freeSpotNear`
([enemy.js:797](../src/game/enemy.js#L797)) places them by collision probe, and
a doorway is "free" by that test. A brood killed in a doorway can currently
scatter children into the corridor.

### How to prove it

Bake a room, drive an enemy at a doorway with the player standing beyond it, and
assert the enemy's position stays within the room's bounds. Then the
counterfactual on current code. Add the brood-in-a-doorway case as its own
assertion — it is a different code path.
